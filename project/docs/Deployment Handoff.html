Taking Field Events Live to a real deployment

The prototype simulates real-time sync with browser storage. This doc maps each piece of the prototype onto a minimal, single-deployable production stack and lists what to build, in order.

What the prototype already settles

  • Domain logic — app/marks.js is plain framework-free JS and ports directly to a Node backend: unit conversion (meters canonical, quarter-inch imperial rounding), best-mark ranking with second-best tiebreak, high-jump countback (misses at best height, then total misses), three-consecutive-miss elimination, and rotation order incl. finals reversal.
  • UI — two React views (officials' entry, spectator board) plus shared components in app/shared-ui.jsx. These translate 1:1 into a Vite + React app.
  • State shape — the store in app/store.js defines the full data model (below). Keep it; it becomes your DB schema and WebSocket payload.

Data model (mirrors the prototype store)

meets    (id, name, venue, date)
events   (id, meet_id, name, kind 'horizontal'|'vertical', record_m)
flights  (id, event_id, name, rounds, cur_height_idx)
heights  (flight_id, idx, meters)            -- vertical only
athletes (id, flight_id, name, team, bib, seed_m, pr_m, sb_m, ord)
attempts (flight_id, athlete_id, round, kind 'mark'|'foul'|'pass', meters)   -- horizontal
bar_attempts (flight_id, athlete_id, height_idx, seq 'XO'|'P'|...)           -- vertical

Store marks in meters, always. Convert at render time only — exactly as marks.js does. Never store the formatted string.

API + socket surface

POST /api/flights/:id/marks        { athleteId, round, kind, meters? }   (officials, authed)
POST /api/flights/:id/bar          { athleteId, result: 'O'|'X'|'P' }
POST /api/flights/:id/bar-height   { heightIdx }                          (set / raise bar)
GET  /api/flights/:id              full flight snapshot (initial load)

socket: join('flight:<id>')  →  server emits 'flight:update' with the
same snapshot after every accepted write. Spectator clients re-render
from the snapshot — identical to the prototype's storage-event path.
  • Snapshots, not diffs. A flight snapshot is <10 KB; sending the whole thing after each mark removes an entire class of sync bugs. (The prototype proves this works — it re-reads full state per change.)
  • Server is authoritative for ranking and elimination. Run the same marks.js functions server-side and include computed places in the snapshot so all clients agree.
  • Auth: a per-meet officials' PIN issued by the meet director is enough for v1. Spectator routes are public and read-only.

Port order

  1. Scaffold Vite + Express + Socket.io; serve the built app from Express.
  2. Move app/marks.js to a shared lib/ imported by both server and client. Add unit tests for ranking, countback, elimination, and imperial rounding first — they encode the competition rules.
  3. Replace app/store.js internals: same mutator names (recordMark, vertRecord, setBar…) but POSTing to the API; replace the storage listener with the socket subscription. The React components shouldn't need to change.
  4. SQLite persistence + snapshot endpoint.
  5. CSV export: one GET /api/flights/:id.csv route — flatten the snapshot; PDF can be print CSS on the spectator view.
  6. Hardening: officials' PIN, optimistic UI with rollback, offline queue for the entry pad (stadium Wi-Fi dies; queue writes in IndexedDB and flush on reconnect).
Two prototype behaviors to keep on purpose: the 2-second polling fallback on the spectator view (belt-and-suspenders when a socket silently dies), and the "tap any cell to correct" pattern on the officials' sheet — corrections are constant at real meets and burying them behind an edit mode is the most common scoring-app mistake.

Known prototype limitations

  • Sync is per-browser (localStorage) — two different phones don't share state until the backend exists.
  • No auth; anyone with the officials' URL can score.
  • CSV/PDF export was descoped from the prototype (per your call) — it's step 5 above.
  • Tie places display correctly (shared place numbers) but award-position resolution (jump-off prompts) is not modeled.
